# Examples: Template-driven Init, Workspaces, Agents, And Staging

Status: draft  
Date: 2026-05-06

This document shows the desired user workflows after the Copier/template refactor in [`REFACTOR.md`](./REFACTOR.md). The examples are intentionally concrete, but the architecture rule is strict: `angee` must not hardcode Django, React, Vite, `manage.py`, pnpm, uv, or any framework-specific port names.

Templates declare those details. The Go CLI only understands generic concepts:

- Template refs: `stacks/dev`, `workspaces/feature-dev`, `agents/claude-code`, `stacks/staging-docker`.
- Copier answers: `--set key=value`.
- Secrets: declared by templates, supplied or generated by Angee.
- Port leases: named by templates, allocated by Angee.
- Sources: git worktrees, clones, archives, storage mounts, templates.
- Services, jobs, and workflows: named declarations with cwd, env, dependencies, readiness checks, and activities.
- Runtime outputs: rendered files, `$ANGEE_ROOT/angee.yaml`, `$ANGEE_ROOT/state/*`, backend files such as `docker-compose.yaml`.

## Design Rules

1. Init commands are noun-first: `angee stack init`, `angee workspace init`, and `angee agent init`.
2. `angee init` is shorthand for the default stack init, normally `angee stack init dev` in a project with `templates/stacks/dev`.
3. `$ANGEE_ROOT/angee.yaml` is the only Angee manifest. There is no separate `.angee/project.yaml`.
4. Runtime-specific commands live in templates or runtime template fragments, not in Go code.
5. Reasonable defaults live in templates. The CLI can auto-detect and allocate, but must not know what `web`, `ui`, or `worker` mean.
6. Generated and derived secrets never go into `.copier-answers.yml`, `$ANGEE_ROOT/angee.yaml`, or backend files such as `docker-compose.yaml`.
7. Named ports are stable within a template, but their numeric values are allocated per stack/workspace/agent to avoid collisions.
8. A workspace without an agent renders under `$ANGEE_ROOT/workspaces/<name>/`; a workspace with an agent template renders under `$ANGEE_ROOT/agents/<name>/`.

## Secret Flag Forms

Recommended target forms:

```sh
--secret anthropic-api-key=sk-ant-...
--secret anthropic-api-key=env:ANTHROPIC_API_KEY
--secret smtp-password=file:./secrets/smtp-password.txt
```

Rules:

- `--secret name=value` supplies the literal value.
- `--secret name=env:VAR` reads from the current environment.
- `--secret name=file:path` reads from a local file.
- Missing generated secrets are generated automatically.
- Missing derived secrets are computed from other secrets and Copier answers.
- Missing required external secrets prompt interactively unless `--yes` is set; with `--yes`, they fail clearly.

## Example 1: Bootstrap `examples/angee-notes` Dev In One Command

The user should be able to enter the example app and bootstrap a complete local dev environment:

```sh
cd ../django-angee/examples/angee-notes
angee stack init dev --yes
```

Equivalent explicit form:

```sh
angee init --yes
```

What happens:

1. Angee resolves `templates/stacks/dev` from the current repo, first-party template library, user template library, or explicit `--template` override.
2. Copier renders the stack template into the current directory.
3. Angee reads the template's `_angee` metadata.
4. Angee generates missing secrets such as app secret keys and writes them to `.angee/.env`.
5. Angee allocates named ports declared by the template, for example `web` and `ui`, and writes `.angee/state/ports.json`.
6. Angee creates runtime data directories such as `.angee/data/` because the template declared them, not because the CLI knows the framework.
7. The operator runs the template-declared post-init workflow, such as dependency sync, migration, and fixture/assets loading.
8. Angee writes `.angee/angee.yaml` and `.copier-answers.yml`.

Resulting tree:

```text
examples/angee-notes/
  .copier-answers.yml
  .angee/
    angee.yaml
    .env
    data/
      staticfiles/
      mediafiles/
      logs/
      tmp/
    state/
      ports.json
```

Then the user starts the dev loop:

```sh
angee dev
```

`angee dev` does not know how to run Django or React. It runs the operator runtime in-process, loads `.angee/angee.yaml`, reads declared services and jobs, resolves `${ports.<name>}` and `${secret:<name>}` references, and reconciles the stack.

Example rendered `.angee/angee.yaml` shape:

```yaml
version: 1

template:
  active: stacks/dev
  source: local:templates/stacks/dev
  answers_file: .copier-answers.yml

state:
  ports_file: .angee/state/ports.json

runtime:
  adapter: process-graph

services:
  web:
    runtime: local
    source: app
    cwd: .
    command: ["uv", "run", "python", "manage.py", "runserver", "127.0.0.1:${ports.web}"]
    after: [migrate]
  ui:
    runtime: local
    source: app
    cwd: ui/react/web
    command: ["pnpm", "dev", "--host", "127.0.0.1", "--port", "${ports.ui}"]
    after: [web]

jobs:
  build:
    kind: process
    source: app
    cwd: .
    command: ["uv", "run", "python", "manage.py", "angee", "build", "--watch"]
    ready:
      stdout: "angee build --watch: ready"
```

The concrete commands are real for `angee-notes`, but they are data. Another template can declare Rust, Node, Rails, or anything else using the same service/job/workflow schema.

## Example 2: Create A Feature Workspace With A Worktree And Dedicated Ports

From the stack root:

```sh
cd ../django-angee/examples/angee-notes
angee workspace init feat-refactor-2 --branch feat-refactor-2 --yes
```

Equivalent explicit form:

```sh
angee workspace init feat-refactor-2 \
  --template workspaces/feature-dev \
  --branch feat-refactor-2 \
  --yes
```

What happens:

1. Angee reads `.angee/angee.yaml` to find the default workspace template, for example `workspaces/feature-dev`.
2. The workspace template declares sources. For this example, it declares a git worktree for the app repo and optionally a companion framework repo using `same-name` branch strategy.
3. Angee allocates a new port set for this workspace from template-declared bands.
4. Copier renders workspace helper files such as `CLAUDE.md`, `.envrc`, and workspace-local `.gitignore`.
5. Angee materializes git sources into the workspace.
6. The operator runs template-declared provisioning steps. If one step is `angee stack init dev --yes` inside `code/`, that command still resolves `templates/stacks/dev`; no framework-specific behavior is baked into the outer CLI.
7. Angee writes workspace state.

Resulting tree:

```text
examples/angee-notes/
  .angee/
    workspaces/
      feat-refactor-2/
        workspace.yaml
        state/
          ports.json
          sources.json
        secrets.env
        CLAUDE.md
        .envrc
        code/
          .git/                  # git worktree metadata
          .angee/
            angee.yaml           # dev stack for this worktree
            .env
            state/ports.json
            data/
```

Run that workspace:

```sh
cd .angee/workspaces/feat-refactor-2/code
angee dev
```

Override ports only when needed. Port names come from the template:

```sh
angee workspace init feat-refactor-2 \
  --branch feat-refactor-2 \
  --port web=8122 \
  --port ui=5192 \
  --yes
```

The CLI does not know `web` means a Django service or `ui` means a Vite service. It only asks the operator to validate that the names exist in `port_leases` and that the numeric ports are free.

## Example 3: Create An Agent Workspace On A Branch With Dedicated Ports

Create an agent-backed workspace. Because an agent template is supplied, the target root is `.angee/agents/<name>/` instead of `.angee/workspaces/<name>/`:

```sh
cd ../django-angee/examples/angee-notes
angee agent init feat-refactor-2 \
  --branch feat-refactor-2 \
  --template agents/claude-code \
  --workspace-template workspaces/feature-dev \
  --secret anthropic-api-key=env:ANTHROPIC_API_KEY \
  --yes
```

What happens:

1. The workspace template creates the code worktree, companion worktrees, ports, and runtime data.
2. The agent template renders agent instructions and runtime config.
3. Angee writes secrets required by the agent runtime, such as the provider API key, into agent-local secret storage.
4. Angee writes `agent-env.json` so the agent can discover its branch, workspace path, port URLs, and MCP endpoints without querying central state.
5. Angee starts any template-declared services if `--start` is passed, or prints the commands to start them.

Resulting tree:

```text
examples/angee-notes/
  .angee/
    agents/
      feat-refactor-2/
        agent.yaml
        secrets.env
        state/
          ports.json
          agent-env.json
          sources.json
        workspace/
          AGENTS.md
          code/
            .angee/
              angee.yaml
              .env
              state/ports.json
              data/
```

Example `agent-env.json`:

```json
{
  "agent_name": "feat-refactor-2",
  "branch": "feat-refactor-2",
  "workspace_root": ".angee/agents/feat-refactor-2/workspace",
  "code_root": ".angee/agents/feat-refactor-2/workspace/code",
  "ports": {
    "web": 8128,
    "ui": 5198
  },
  "urls": {
    "web": "http://127.0.0.1:8128",
    "ui": "http://127.0.0.1:5198"
  },
  "mcp_servers": {
    "app": { "transport": "streamable-http", "url": "http://127.0.0.1:8128/mcp/" },
    "files": { "transport": "stdio", "command": ["node", "...", "/workspace"] }
  }
}
```

Start the agent:

```sh
angee agent start feat-refactor-2
```

Or run the dev services manually:

```sh
cd .angee/agents/feat-refactor-2/workspace/code
angee dev
```

## Example 4: Install A Docker Staging Environment For The Whole Platform

A staging stack should be a normal stack template, not a branch in Go code:

```sh
cd ../django-angee/examples/angee-notes
angee stack init staging-docker \
  --set domain=staging.notes.example.com \
  --set image_tag=ghcr.io/fyltr/angee-notes:2026-05-06 \
  --secret anthropic-api-key=env:ANTHROPIC_API_KEY \
  --secret smtp-password=env:SMTP_PASSWORD \
  --yes
```

Then start it:

```sh
angee up
```

What the staging template declares:

- Docker services for the app, workers, scheduler, database, cache, object storage, operator, reverse proxy, and agent runtimes.
- Generated secrets such as database passwords, app secret keys, service-account tokens, and OpenBao/bootstrap tokens.
- Required external secrets such as AI provider keys, SMTP passwords, OAuth client secrets, or deploy credentials.
- Derived secrets such as database URLs and broker URLs.
- Domains and TLS settings as Copier answers.
- Build/image policy as Copier answers.
- Post-init workflows such as migrations and assets loading.

Example staging template metadata:

```yaml
_min_copier_version: "9.0"
_subdirectory: template
_templates_suffix: .jinja

_angee:
  schema: 1
  kind: stack
  name: staging-docker

  secrets:
    - { name: app-secret-key, generated: true, length: 50 }
    - { name: db-password, generated: true, length: 32 }
    - { name: broker-password, generated: true, length: 32 }
    - { name: database-url, derived: "postgresql://app:${db-password}@postgres:5432/{{ project_name }}" }
    - { name: broker-url, derived: "redis://:${broker-password}@redis:6379/0" }
    - { name: anthropic-api-key, required: true }
    - { name: smtp-password, required: true }

  compose:
    input: angee.yaml
    output: docker-compose.yaml

  workflows:
    post_init:
      - { activity: run-job, job: migrate }
      - { activity: run-job, job: fixtures }

project_name:
  type: str
  default: angee-notes

domain:
  type: str
  default: localhost

image_tag:
  type: str
  default: latest
```

Example rendered `angee.yaml.jinja` fragment:

```yaml
name: {{ project_name }}-staging

services:
  web:
    image: {{ image_tag }}
    lifecycle: platform
    domains:
      - host: {{ domain }}
        port: 8000
        tls: true
    env:
      DATABASE_URL: "${secret:database-url}"
      SECRET_KEY: "${secret:app-secret-key}"

  worker:
    image: {{ image_tag }}
    lifecycle: worker
    command: celery -A runtime worker -l info
    env:
      DATABASE_URL: "${secret:database-url}"
      BROKER_URL: "${secret:broker-url}"

  postgres:
    image: pgvector/pgvector:pg17
    lifecycle: sidecar
    env:
      POSTGRES_PASSWORD: "${secret:db-password}"
```

The Go CLI does not know these are Django, Celery, Postgres, Redis, or Traefik concepts. It asks the operator to render the template, resolve secrets, compile deployment backend files, and reconcile the stack.

## Example 5: Update A Stack Template

Use `angee stack update` to refresh the current stack from its active template:

```sh
angee stack update
```

Update to a specific template ref:

```sh
angee stack update --ref v4
```

Change stack target, then update from that template:

```sh
angee stack switch staging-docker
```

`stack switch` should be equivalent to:

```sh
angee stack set-template stacks/staging-docker
angee stack update
```

What happens:

1. Angee reads `.angee/angee.yaml.template.active`.
2. Angee resolves the template and any `_angee.extends` chain.
3. Copier runs its update algorithm against `.copier-answers.yml`.
4. The operator re-runs post-render steps: secrets are preserved, new generated secrets are added, ports are preserved unless explicitly changed, compose is recompiled.
5. Conflicts are left as Copier conflicts for the user to resolve.

## Example 6: Override Defaults Without Hardcoding CLI Flags

Avoid runtime-specific flags such as `--django-port` or `--vite-port`. Use template-declared names:

```sh
angee stack init dev \
  --port web=8120 \
  --port ui=5190 \
  --set project_name=notes-local \
  --yes
```

Avoid runtime-specific secret flags. Use template-declared names:

```sh
angee stack init staging-docker \
  --secret anthropic-api-key=env:ANTHROPIC_API_KEY \
  --secret smtp-password=env:SMTP_PASSWORD \
  --yes
```

Avoid runtime-specific process flags. Put service/job overrides into a template or into `.angee/angee.yaml`:

```yaml
services:
  ui:
    cwd: ui/react/web
    command: ["pnpm", "dev", "--host", "0.0.0.0", "--port", "${ports.ui}"]
```

## What Must Not Be Hardcoded In `angee`

The Go CLI must not contain knowledge like:

- `django` means `manage.py`.
- `react` means Vite.
- `web` defaults to port `8100`.
- `ui` defaults to port `5173`.
- `uv sync` should run before a server starts.
- `pnpm dev` is the frontend command.
- `.angee/data/staticfiles` is required for a specific framework.
- `fixtures load` or `assets load` is the right post-init command.

Those belong in templates, runtime template fragments, or the rendered `.angee/angee.yaml` service/job declarations.

The Go CLI may know how to:

- Resolve and render Copier templates.
- Validate `_angee` metadata schema.
- Allocate named ports.
- Resolve generated, derived, and supplied secrets.
- Materialize declared sources.
- Ask the operator to run declared services, jobs, and workflows.
- Compile `angee.yaml` when a template rendered it.
- Apply a selected runtime backend such as Docker Compose.

## Minimal Template Contract

A template that wants `angee stack init dev` to work should declare enough metadata for Angee to be generic:

```yaml
_angee:
  schema: 1
  kind: stack
  name: dev

  data_dirs:
    - .angee/data/staticfiles
    - .angee/data/mediafiles
    - .angee/data/logs
    - .angee/data/tmp

  port_leases:
    - { name: web, default: 8100, band: app, export_env: WEB_PORT }
    - { name: ui, default: 5173, band: ui, export_env: UI_PORT }

  secrets:
    - { name: app-secret-key, generated: true, length: 50 }

  services:
    web:
      command: ["uv", "run", "python", "manage.py", "runserver", "127.0.0.1:${ports.web}"]
    ui:
      cwd: ui/react/web
      command: ["pnpm", "dev", "--port", "${ports.ui}"]

  jobs:
    build:
      command: ["uv", "run", "python", "manage.py", "angee", "build", "--watch"]

project_name:
  type: str
  default: "{{ _folder_name }}"
```

This template happens to run a Django plus React app, but Angee sees only declared sources, volumes, port leases, generated secrets, services, jobs, and workflows.
